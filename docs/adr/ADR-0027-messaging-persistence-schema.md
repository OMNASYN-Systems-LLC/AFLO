# ADR-0027: Secure-messaging persistence schema (conversation threads + messages)

## Status

**Accepted** ‚Äî 2026-07-20 (Production Cutover directive, PHASE 10 ‚Äî persistent
messaging)

## Context

Staff‚Üîclient secure messaging (ADR era: the messaging store wiring, interactive
send, and read receipts) shipped as domain logic (`@aflo/shared`
`ConversationThread`/`Message`, projected client-safe via `toClientThreadView`)
and a deterministic kernel (`@aflo/rules` `messaging.v1.0.0`), but with **no
persistence** ‚Äî threads and messages lived only in the in-memory store, so a
conversation could not survive a redeploy. The cutover directive (PHASE 2/10)
requires the messaging tables in Neon, kept as a **separate** migration from the
auth persistence schema (0005) for reviewability.

## Decision

Forward-only migration **0006** (offline-generated by `drizzle-kit generate`; no
credentials) adds two tenant-owned tables with the schema's existing conventions
(uuid PKs, `timestamps`, FK `references`, org-scoped indexes):

- **`conversation_threads`** ‚Äî org-scoped; `client_id`, `subject`, `status`
  (`thread_status`: open | closed), nullable `last_message_at`.
- **`messages`** ‚Äî org-scoped; `thread_id`, `client_id`, `sender_role`
  (`message_sender_role`: staff | client), a **polymorphic** `sender_id` (an
  `organization_members.id` for staff, a `clients.id` for the client,
  discriminated by `sender_role` ‚Äî no FK, same pattern the schema uses for the
  pre-table `ai_run_id` columns), `body_encrypted`, `sent_at`, and the two read
  receipts `read_by_client_at` / `read_by_staff_at`.

Two new enums are **kernel-owned** ‚Äî built from the `@aflo/rules` messaging
constants `THREAD_STATUSES` and `MESSAGE_SENDER_ROLES` (added here, with the
existing `ThreadStatus`/`MessageSenderRole` types re-derived from them so the
change is type-identical). The database enum and the deterministic messaging
kernel are now the same source and can never disagree; a lockstep test asserts
the derivation held.

### Message bodies are ciphertext only (the key safety call)

`messages` has **no plaintext `body` column**. The body is stored solely as
`body_encrypted` ‚Äî the schema's `encrypted` `bytea` custom type ‚Äî so the database
holds only application-layer ciphertext. The repository (next slice) encrypts on
write and decrypts on read; the DB, its backups, and its query surface never see
a readable body. A static test asserts a plaintext `body` column is absent and
`body_encrypted` is `bytea NOT NULL`; a runtime test proves the stored bytes are
the exact opaque ciphertext round-tripped (never plaintext).

Internal staff notes remain a **separate** table (`notes`) and are not
representable here, preserving the structural boundary that keeps an internal
note out of any client thread view. Message bodies must also never be copied into
the outbox payload (delivery metadata only) ‚Äî enforced by the producer/repository
in the wiring slice, not by this DDL.

### RLS boundary

Both tables carry `organization_id` and get `ENABLE` + `FORCE` RLS with the
migration-0003 fail-closed `org_isolation` policy
(`"organization_id" = nullif(current_setting('app.current_org_id', true),
'')::uuid`). Unlike the auth-resolver tables (0005), there is nothing that reads
messaging before an org context exists, so no exemption applies ‚Äî the derived
tenant-set coverage test (`rls.test.ts`) requires the RLS DDL and would fail CI
if it were missing.

## Consequences

- **Proven credential-free on PGlite** (`rls-messaging.test.ts`, non-superuser
  role): thread + message list/direct-id org isolation, cross-tenant write
  rejection, fail-closed on unset AND empty-string context, a read-receipt
  UPDATE touching zero foreign-org rows (RLS hides the row), and the
  ciphertext-only body round-trip. Plus the enum lockstep and static
  table-invariant tests. **108 database tests total.** Additive-only ‚Äî no
  destructive statements.
- **Application to Neon is credential-gated** (PHASE 3/4). GitHub remains the
  migration authority: 0006 is committed and ready; applying it (dev ‚Üí verify ‚Üí
  preview ‚Üí verify, via the repo's `db:migrate` workflow with `DATABASE_URL`
  supplied by the deploy pipeline) is the founder/deploy step. No manual Neon
  DDL. **Neon `main` is not touched.**
- **Not yet wired.** The Drizzle repositories over these tables ‚Äî scoped through
  `withOrgContext` (ADR-0025), encrypting the body, and keeping message bodies
  out of the outbox ‚Äî are the follow-up slice, alongside the auth-resolver
  repositories (0005's PHASE-5 prerequisites in ADR-0026 still apply).

## Post-review note / repository-slice follow-up

An adversarial review (SAFE TO MERGE; no CRITICAL/HIGH/MEDIUM) flagged one
LOW/informational item to carry into the repository slice: there is no DB-level
`CHECK` that a `messages` row's `organization_id` (and `client_id`) matches its
referenced `conversation_threads` row's. This is **not** a cross-tenant leak ‚Äî a
message carries its own `organization_id`, so RLS scopes it to the inserting org
regardless, and a foreign thread is invisible to that org; the only possible
effect is a dangling reference *inside a single org's own data*. It matches the
repo-wide pattern (no cross-table org-consistency constraints anywhere) and is
correctly a repository responsibility: the `withOrgContext`-scoped
`MessageRepository` MUST verify the target thread belongs to the current org (and
the same client) before inserting, since FK validation bypasses RLS.
