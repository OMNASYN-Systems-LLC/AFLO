# ADR-0026: Auth persistence schema (identity bridge)

## Status

**Accepted** — 2026-07-20 (Production Cutover directive, PHASE 2 — auth
persistence schema)

## Context

The identity work (ADR-0018…0024) shipped as domain logic and contracts with no
persistence, so Clerk identity, invitations, client links, webhook receipts, and
session revocation had nowhere durable to live — the founder's cutover review
correctly flagged that Neon carried migrations through `0004` but none of these
tables. Without them the authenticated loop cannot survive a redeploy.

## Decision

Forward-only migration **0005** adds five tables (offline-generated by
`drizzle-kit generate`; no credentials) with the schema's existing conventions
(uuid PKs, `timestamps`, FK `references`, unique/partial-unique indexes) and
digest-only secret storage:

- **`identity_provider_accounts`** — `(provider, provider_user_id)` unique →
  `aflo_user_id`. The Clerk→AFLO user mapping.
- **`invitations`** — org-scoped; `invitation_type`, `invited_role` (the four
  invitable @aflo/auth roles), `intended_client_id`, `token_digest` (sha256 hex,
  never the raw token), lifecycle timestamps. Unique `(org, token_digest)` and a
  partial-unique `(org, email) WHERE status='pending'`.
- **`client_user_links`** — org-scoped; partial-unique
  `(org, client_id) WHERE active` and `(org, user_id) WHERE active` enforce
  "a client is linked to exactly one active user, and vice versa."
- **`provider_webhook_events`** — `(provider, provider_event_id)` unique
  (Svix-id idempotency); `payload_digest` only, never the payload or secret.
- **`session_revocations`** — user-scoped, nullable org; `reason_code`,
  `provider_session_id_digest` (nullable = revoke all), optional expiry.

### RLS boundary (the key design call)

`invitations` and `client_user_links` carry `organization_id` and get `ENABLE` +
`FORCE` RLS with the migration-0003 fail-closed `org_isolation` policy. The other
three are **intentionally NOT org-RLS-scoped**, exactly like
`organizations`/`users`/`rule_versions`: they are read by the auth resolver
BEFORE an org context exists (a user's identity mapping, their revocations) or
across orgs (provider webhook receipts). RLS-scoping a nullable-org table would
hide null-org rows from the resolver and break disable/revoke. Access to them is
the privileged auth-resolver/service path.

The `rls.test.ts` invariant now carries an explicit `RLS_EXEMPT_TENANT_TABLES`
allowlist (`session_revocations`) so the exemption is deliberate and documented,
not an oversight.

## Consequences

- **Proven credential-free on PGlite** (`rls-auth-tables.test.ts`, non-superuser
  role): invitations list/direct-id isolation, cross-tenant write rejection,
  fail-closed-on-unset, invitation token + pending-email uniqueness, and a
  client-user double-claim rejection. Plus the existing migration-integrity and
  RLS static tests (94 database tests total). Additive-only — no destructive
  statements.
- **Application to Neon is credential-gated** (PHASE 3/4). GitHub remains the
  migration authority: this migration is committed and ready; applying it to Neon
  dev then preview (via the repo's `db:migrate` workflow, `DATABASE_URL` supplied
  by the deploy pipeline) is the founder/deploy step. No manual Neon DDL.
- **Not yet wired.** The repositories over these tables (PHASE 5) and the accept
  path's token-scoped invitation lookup (a `SECURITY DEFINER` function, since the
  accept-by-token read precedes org context) are follow-ups.
- **Messaging persistence** (`conversation_threads`, `messages`) is the very next
  migration (0006), kept separate for reviewability.
